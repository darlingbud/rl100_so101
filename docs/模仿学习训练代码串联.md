# RL-100 第一步模仿学习训练代码串联

本文梳理 RL-100 训练流水线中第一步 **Behavior Cloning（BC，行为克隆/模仿学习）** 的代码调用关系。重点以仓库中较典型的 **3D 点云 + Diffusion（DDIM）策略** 为例，说明从 Shell 启动脚本、Hydra 配置、数据集，到模型损失、反向传播和 checkpoint 保存的完整链路。

> 说明：仓库中的两阶段启动脚本把“Stage 1”定义为“生成 BC、IQL critic 和 dynamics 等基础产物”。其中，纯模仿学习只是 `train.py` / `train_ddp.py` 内部标记为 `stage 1-1` 的第一段。本文主要讲这段 BC 训练，不把后续 critic、value 和 dynamics 训练混入模仿学习。


## 2. 启动入口

### 2.1 推荐从两阶段脚本进入

3D Diffusion 的两阶段入口是：

```text
scripts/Diffusion/Offline/3D/train_policy_two_stage.sh
```

示例命令：

```bash
bash scripts/Diffusion/Offline/3D/train_policy_two_stage.sh \
  rl100 adroit_door_medium 0112 100 4
```

五个位置参数依次是：

| 参数 | 示例 | 含义 |
|---|---|---|
| `alg_name` | `rl100` | 算法/实验标识，主要参与实验名拼接 |
| `task_name` | `adroit_door_medium` | Hydra task 配置名 |
| `addition_info` | `0112` | 额外实验标签 |
| `seed` | `100` | 随机种子 |
| `NUM_GPUS` | `4` | Stage 1 使用的 GPU 数量，默认 4 |

脚本默认设置：

```bash
config_name='rl100_3d_epsilon'
STAGE1_ENTRY=train_ddp.py
STAGE1_BPPO_STEPS=0
```

因此，Stage 1 最终会展开为类似下面的命令：

```bash
cd RL-100
torchrun --standalone --nproc_per_node=4 train_ddp.py \
  --config-name=rl100_3d_epsilon.yaml \
  task=adroit_door_medium \
  only_bc=True \
  offline=True \
  policy.scheduler_type=ddim \
  unio4.bppo_steps=0 \
  training.num_epochs=600 \
  dataloader.batch_size=1024 \
  optimizer.lr=2.83e-4 \
  ...
```

`train_policy_two_stage.sh` 中的 `get_common_params()` 是实际训练参数的主要来源；这些命令行参数会覆盖 YAML 中的默认值。

如果只想运行脚本定义的 Stage 1、暂时不启动 Stage 2 的 BPPO 参数扫描，可以使用：

```bash
RUN_SWEEP=false bash scripts/Diffusion/Offline/3D/train_policy_two_stage.sh \
  rl100 adroit_door_medium 0112 100 4
```

注意：这仍会生成脚本所要求的 BC、critic/value 和 dynamics 产物，并非只执行 BC 主循环后立即退出。

### 2.2 Python 主入口

单卡与多卡入口分别是：

```text
RL-100/train.py
RL-100/train_ddp.py
```

二者底部都通过 Hydra 注册主函数：

```python
@hydra.main(
    version_base=None,
    config_path=str(pathlib.Path(__file__).parent / "rl_100" / "config")
)
def main(cfg):
    workspace = TrainDP3Workspace(cfg)
    workspace.run()
```

也就是说，Python 入口本身很薄，核心调度逻辑位于 `TrainDP3Workspace`。

## 3. Hydra 配置如何合并

### 3.1 主配置

3D Diffusion 示例使用：

```text
RL-100/rl_100/config/rl100_3d_epsilon.yaml
```

文件开头声明默认任务：

```yaml
defaults:
  - task: adroit_hammer
```

启动脚本传入：

```text
task=adroit_door_medium
```

于是默认任务会被替换为：

```text
RL-100/rl_100/config/task/adroit_door_medium.yaml
```

最终配置的优先级可以概括为：

```text
主 YAML 默认值 < task YAML < Shell 命令行覆盖参数
```

### 3.2 与 BC 直接相关的关键配置

| 配置 | 作用 |
|---|---|
| `policy._target_` | 指定 Hydra 实例化的策略类，本例是 `RL1003D` |
| `task.dataset._target_` | 指定数据集类，本例是 `AdroitDataset` |
| `task.dataset.zarr_path` | 示范数据路径 |
| `horizon` | 每个训练样本包含的动作/轨迹长度 |
| `n_obs_steps` | 输入模型的历史观测帧数 |
| `n_action_steps` | 执行或评估时取出的动作步数 |
| `policy.scheduler_type` | `ddim` 表示采用扩散噪声预测式 BC；也可配置为 `flow` |
| `training.num_epochs` | BC 训练轮数 |
| `dataloader.batch_size` | BC batch size |
| `optimizer` | 优化器类型及学习率 |
| `training.use_ema` | 是否维护模型参数的指数滑动平均副本 |
| `checkpoint.save_ckpt` | 是否周期性保存 checkpoint |
| `only_bc` | checkpoint 时额外保存 BC policy；当前代码不会因此直接跳过后续 critic/dynamics |

## 4. Workspace 初始化

核心类位于：

```text
RL-100/train.py → class TrainDP3Workspace
```

构造函数主要完成以下工作：

```python
self.model = hydra.utils.instantiate(cfg.policy)
self.ema_model = copy.deepcopy(self.model)       # use_ema=True 时
self.optimizer = hydra.utils.instantiate(
    cfg.optimizer,
    params=self.model.parameters()
)
```

对应关系是：

```text
cfg.policy
  → rl_100.policy.rl100_3d.RL1003D

cfg.optimizer
  → torch.optim.AdamW(model.parameters())
```

构造函数还会固定 PyTorch、NumPy 和 Python 随机种子，并根据配置初始化后续离线 RL 使用的 `BehaviorProximalPolicyOptimization`。但后者不是 BC 每个 batch 的训练执行者。

## 5. 示范数据加载链路

### 5.1 数据集配置

`adroit_door_medium.yaml` 中定义：

```yaml
dataset:
  _target_: rl_100.dataset.adroit_dataset.AdroitDataset
  zarr_path: data/adroit_door_medium.zarr
  horizon: ${horizon}
  pad_before: ${eval:'${n_obs_steps}-1'}
  pad_after: ${eval:'${n_action_steps}-1'}
  val_ratio: 0.02
```

由于启动脚本已经执行 `cd RL-100`，相对路径实际指向：

```text
RL-100/data/adroit_door_medium.zarr
```

### 5.2 Workspace 实例化数据集

`TrainDP3Workspace.run()` 中执行：

```python
dataset = hydra.utils.instantiate(cfg.task.dataset)
train_dataloader = DataLoader(dataset, **cfg.dataloader)

normalizer = dataset.get_normalizer()
val_dataset = dataset.get_validation_dataset()
val_dataloader = DataLoader(val_dataset, **cfg.val_dataloader)

self.model.set_normalizer(normalizer)
self.ema_model.set_normalizer(normalizer)
```

这里完成了四件事：

1. 从 Zarr 读取示范轨迹；
2. 用 `SequenceSampler` 把轨迹切成固定 `horizon` 的序列；
3. 按 `val_ratio` 划分训练集和验证集；
4. 从数据统计量构造 `LinearNormalizer` 并注入策略。

### 5.3 一个 batch 的结构

`AdroitDataset.__getitem__()` 返回的主要字段为：

```text
batch
├── obs
│   ├── point_cloud     # [B, T, N, C]
│   ├── agent_pos       # [B, T, D_state]
│   └── image           # [B, T, C, H, W]
├── action              # [B, T, D_action]
├── next_obs
├── next_action
├── reward
├── not_done
└── return
```

对纯 BC 来说，最核心的字段是 `obs` 和示范动作 `action`；奖励、回报等字段主要供后续离线 RL 使用。

## 6. 策略模型构建

本例策略类位于：

```text
RL-100/rl_100/policy/rl100_3d.py → class RL1003D
```

它大体包含：

```text
观测编码器 obs_encoder
  输入：点云、机器人状态（配置允许时也可包含图像）
  输出：观测条件特征 global_cond

扩散策略网络 model
  输入：带噪动作轨迹、扩散时间步、观测条件
  输出：预测噪声 epsilon（本例 prediction_type=epsilon）

噪声调度器 ddim_scheduler
  负责前向加噪和推理阶段的逐步去噪
```

当前两阶段 3D Diffusion 脚本还覆盖了：

```text
policy.encoder_type=dp3vib
policy.model=skipnet
policy.use_vib=True
policy.use_recon=True
policy.scheduler_type=ddim
```

因此观测编码器除为策略提供条件特征外，还会产生 VIB KL 和重建损失。

## 7. BC 损失如何计算

训练循环调用：

```python
raw_loss, loss_dict = self.model.compute_loss(batch)
```

实际实现位于：

```text
RL-100/rl_100/policy/rl100_3d.py → RL1003D.compute_loss()
```

DDIM 模式下的主要过程如下。

### 7.1 数据归一化

```python
nobs = self.normalizer.normalize(batch['obs'])
nactions = self.normalizer['action'].normalize(batch['action'])
```

当 `no_pre_action=True` 时，还会裁掉历史动作部分：

```python
nactions = nactions[:, self.n_obs_steps - 1:]
```

### 7.2 编码观测条件

策略取前 `n_obs_steps` 帧观测，送进点云/状态编码器：

```python
this_nobs = ...
vib_recon_loss, loss_items, nobs_features = \
    self.obs_encoder.Recon_VIB_loss(this_nobs)
global_cond = nobs_features.reshape(batch_size, -1)
```

### 7.3 对示范动作加噪

将归一化后的示范动作序列视为干净轨迹 `trajectory`：

```python
noise = torch.randn(trajectory.shape, device=trajectory.device)
timesteps = torch.randint(0, num_train_timesteps, (batch_size,))
noisy_trajectory = self.ddim_scheduler.add_noise(
    trajectory, noise, timesteps
)
```

可写成扩散模型常见形式：

```text
x_t = α_t · x_0 + σ_t · ε
```

其中 `x_0` 是示范动作轨迹，`ε` 是随机高斯噪声。

### 7.4 网络预测噪声

```python
pred = self.model(
    sample=noisy_trajectory,
    timestep=timesteps,
    global_cond=global_cond
)
```

当前配置的 `prediction_type=epsilon`，所以监督目标就是刚才采样的噪声：

```python
target = noise
loss = F.mse_loss(pred, target, reduction='none')
```

若使用 `dp3vib` 编码器，最终损失还会加上 VIB/重建联合损失：

```python
loss = diffusion_mse + vib_recon_loss
```

日志中会分别记录：

```text
bc_loss
kl_loss
recon_loss
```

因此，这里的“模仿”不是直接回归动作，而是学习在观测条件下对示范动作轨迹进行去噪；推理时从随机噪声逐步生成动作。

## 8. 反向传播与参数更新

`TrainDP3Workspace.run()` 中的 BC 主循环可以简化为：

```python
for epoch in range(cfg.training.num_epochs):
    for batch in train_dataloader:
        batch = move_to_device(batch)

        raw_loss, loss_dict = self.model.compute_loss(batch)
        loss = raw_loss / gradient_accumulate_every
        loss.backward()

        if self.global_step % gradient_accumulate_every == 0:
            self.optimizer.step()
            self.optimizer.zero_grad()
            lr_scheduler.step()

        if cfg.training.use_ema:
            ema.step(self.model)
```

这一段就是第一步模仿学习的核心闭环：

```text
示范 batch
  → 归一化
  → 观测编码
  → 示范动作加噪
  → 网络预测噪声
  → MSE + VIB/重建损失
  → 反向传播
  → AdamW 更新
  → EMA 更新
```

## 9. 评估、日志与保存

每个 epoch 结束后，Workspace 可能执行：

1. 使用普通模型或 EMA 模型进行环境 rollout；
2. 从固定训练 batch 采样动作，计算 `train_action_mse_error`；
3. 将训练损失、学习率和 rollout 指标写入 WandB；
4. 按 `checkpoint_every` 保存 checkpoint。

主要 checkpoint 路径是：

```text
<hydra.run.dir>/checkpoints/latest.ckpt
```

当 `only_bc=True` 且触发 checkpoint 保存时，还会调用 `unio4.save()`，将 BC policy 保存到：

```text
<hydra.run.dir>/bc/
```

两阶段脚本判断 Stage 1 是否完整时，除了检查：

```text
checkpoints/latest.ckpt
```

还会检查：

```text
Q_bc_20.pt
value_20.pt
saved_models*/dynamics.pth
```

后面三类文件来自 BC 之后的 critic/value/dynamics 阶段，不是 BC 损失训练本身的产物。

## 10. BC 结束后代码会走向哪里

BC 主循环结束后，`TrainDP3Workspace.run()` 不会因为 `only_bc=True` 自动返回，而会继续执行后续初始化，例如：

```python
self.model.set_critic_normalizer(normalizer)
iql, Q_bc, value = self.model.initialize_critic(...)
```

随后进入 critic、value、dynamics，以及由配置控制的离线策略优化逻辑。

因此需要特别注意：

- `only_bc=True` 在当前实现里主要影响 BC policy 的额外保存；
- `unio4.bppo_steps=0` 能关闭 Stage 2 的 BPPO 更新步数；
- 它们都不等价于“BC 完成后立刻退出进程”；
- 两阶段脚本正是利用这一行为，一次性生成 Stage 2 所需的全部基础产物。

如果未来需要一个严格的“纯 BC 训练入口”，比较清晰的实现方式是在 BC checkpoint 保存完成后增加显式配置（如 `bc_only_exit=True`）并提前 `return`，而不是复用目前语义不完整的 `only_bc`。

## 11. 单卡与 DDP 的区别

### 单卡

```text
python train.py
```

模型、优化器和 DataLoader 都由单进程管理，最适合调试调用链。

### 多卡

```text
torchrun --nproc_per_node=N train_ddp.py
```

`train_ddp.py` 会初始化 NCCL process group、设置当前 rank 对应的 CUDA 设备，并使用分布式训练逻辑。总体业务链路仍然相同：

```text
Hydra 配置 → Workspace → Dataset → compute_loss → backward → optimizer → checkpoint
```

阅读代码时建议先看结构更直观的 `train.py`，再对照 `train_ddp.py` 中的数据分片、模型包装、跨 rank 同步和仅 rank 0 保存等差异。

## 14. 最简总结

第一步模仿学习的本质代码链路是：

```text
Shell 选择实验参数
  → Hydra 生成 cfg
  → Workspace 实例化 RL1003D 和 AdroitDataset
  → Dataset 从 Zarr 采样“观测—示范动作”序列
  → RL1003D.compute_loss() 对动作加噪并学习条件去噪
  → Workspace 执行反向传播、AdamW、LR scheduler 和 EMA
  → rollout/采样评估
  → 保存 latest.ckpt 和 BC policy
```

这份 BC checkpoint 随后作为 critic/value/dynamics 训练和 Stage 2 离线 RL 后训练的策略初始化。

执行转换并合并全部子数据集：

```bash
conda run --no-capture-output -n lerobot \
```

```text
  → ro101_clip.yaml
  → RO101Dataset
  → image_front + image_side + agent_pos
  → 2D Diffusion BC
```
